# 記事執筆ワークフロー

**一度の指示で記事を完成させない。** 一気に書くと100%「ネットのまとめ記事」になり、AdSenseの「有用性の低いコンテンツ」判定を招く。以下の4フェーズに分けて進める。

## フェーズ1：構成案（一次情報インプット）

執筆前に、取締役（または著者：大谷一輝）が持っている**一次情報・実体験**を先に聞き出してから構成を作る。一次情報がまだ無い場合は、このフェーズで止めて先に聞く。「後から足せばいい」で構成だけ先に進めない——一次情報が構成の骨格に組み込まれていないと、後付けの体験談は浮いた印象になる。

### 簿記論点マイクロモジュール（`docs/boki/`）を先に確認・作成する（2026-07-15追加）

簿記関連の記事で扱う論点は、`docs/boki/boki{1,2,3}/...`配下に「1ファイル＝1最小論点」のマイクロモジュール（`.md`）として蓄積している。既存ファイルがあれば構成案の一次資料として使う。無ければ、その記事で扱う論点1つにつき1ファイルを新規作成してから執筆に入る（`~/Documents/study-quest-reference/`の教科書PDFから該当箇所を`pdftotext`で抽出し原文を引用する）。

**5,000ページ超の教科書PDF全体を一括でマイクロモジュール化することはしない**（2026-07-15、取締役と合意）。実際に記事で扱う論点だけを、記事執筆のたびに1つずつ増やす。フォーマットは以下の4見出し固定：
1. コア概念（一言で言うと）
2. 標準的な仕訳パターン、または計算プロセス
3. 執筆時の必須キーワード・引っかかりやすい落とし穴
4. テキスト原文・参照データ（教科書からの引用＋本サイトのどの記事で使ったかの参照）

ディレクトリ構成：`boki3/`／`boki2/commercial/`・`boki2/industrial/`／`boki1/commercial-accounting/`・`boki1/industrial-cost/`。ファイル名は半角英数字ケバブケース。

### リサーチログ（`docs/research-logs/`）を必ず残す（2026-07-15追加）

記事の生成・修正でWeb検索または教科書PDF等のドキュメント参照を行ったら、`docs/research-logs/`配下に、対応する記事ファイル名＋`-research.md`でログを残す（ディレクトリ構成は`docs/boki/`と同じ）。フォーマットは固定：

```
# リサーチログ：[記事タイトル]
- 執筆/リサーチ日：[日付]
## 1. 検索クエリ一覧
## 2. 参照したURLとソース情報の要約（サイト名・抽出した事実・注意点や残された論点）
## 3. AIによる補足メモ（数ヶ月後の自分への引き継ぎ）
```

**検索・参照を実際にしていない場合は「なし」と正直に書く。** 検索クエリや参照URLを捏造しない——このログの目的は将来の修正時に「何を根拠にこの記事を書いたか」を正確に追跡できることであり、事実と異なる記録は害悪でしかない。

**後日の修正時：** 対象記事を修正する指示を受けたら、記事ファイルだけでなく対応する`-research.md`・`docs/boki/`のマイクロモジュールも読み込んでから変更箇所を検討する。

## フェーズ2：素材投入（生の声をそのまま受け取る）

構成の各見出しに対応する生のメモ・箇条書き・音声入力の文字起こしをそのまま受け取り、本文の下書きに統合する。この段階では整えすぎない。ニュアンス（失敗談・数字・固有名詞）を削らない。

## フェーズ3：執筆（禁止フレーズを避ける）

以下のようなAI特有の言い回しは使わない：

- 「〜を解説していきます」「〜について見ていきましょう」
- 「結論から申し上げますと」「結論から言うと」
- 「〜が重要です」「〜が大切です」（具体性のない一般論の締め）
- 「〜と言えるでしょう」「〜と言われています」（誰の意見か不明な断定回避）
- 「いかがでしたか？」「参考になれば幸いです」
- 箇条書きの多用による情報の羅列だけの構成

代わりに、著者の一人称の判断・失敗談・具体的な数字を使う。[[project_author_persona_unification]]で統一済みの著者ペルソナ（大谷一輝、関西の大学3回生、日商簿記1級勉強中）の視点を崩さない。

### 「教え方の翻訳者」であること（2026-07-14追加）

資格試験は参考書ごとに「教える中身」は同じ（試験範囲・条文・仕訳ルールは共通）だが、「教え方（噛み砕き方）」だけが差別化要素になる。これはGoogleが最も嫌う「辞書的な正論のコピペ」と、最も評価する「翻訳者としての解説」の違いでもある。執筆時は以下を意識する：

- **専門用語は必ず身近なたとえ話に変換する**：教科書的な定義文をそのまま書かない。日常の道具・体験に置き換える（例：減価償却→「スマホを買った瞬間から年々価値が下がっていくのと同じ」）
- **間違いやすい罠を先回りする**：「正しいやり方」だけでなく、「なぜ自分がやると答えが合わないのか」という初心者がつまずく具体的なポイントを共有する
- **図解を差し込む**：文章だけで説明を終わらせず、手書き図解・比較表・矢印を使った流れ図を1枚入れる。取締役はiPad（GoodNotes等）で日常的に勉強しているため、その手書きノートを図解素材として活用できないか都度確認する
- 判定基準は「教科書通りの正しい言葉で説明していないか？　自分の言葉に翻訳できているか？」の自問

### テキストベースの図解（T字勘定・矢印フロー）も併用する（2026-07-15追加）

SVG画像を作るほどではない細かい仕訳の流れ・勘定連絡は、本文中にテキストだけで図解する。特に工業簿記・原価計算（勘定の振替が多い分野）で有効。

- **T字勘定表現**：`<pre>`や等幅フォントのボックスで、借方・貸方を左右に並べたテキスト図を書く
- **矢印フロー**：「仕掛品 → 完成 → 製品 → 売上原価」のように、勘定の流れを矢印でテキスト表現する
- SVG図解と使い分ける：ページ全体の戦略マップ・複数要素の比較はSVG、1つの仕訳・1つの勘定連絡の流れはテキスト図、が目安

### たとえ話は2段階で出す

つまずきやすい専門用語には、たとえ話を1つだけでなく**タイプの異なる2つ**を段階的に出すと理解度が上がる。1つの文にまとめて詰め込まない（読者が混乱する）。以下の順で構成する：

1. **個人レベルのたとえ**（ミクロ・感覚）：読者の財布・日常の買い物・サブスクなど、誰でも自分ごととして想像できるシンプルな例で全体像をふんわり掴ませる
2. **バイト先・ビジネスレベルのたとえ**（マクロ・実務）：飲食店・デリバリー・コンビニなど、著者自身のアルバイト経験に紐づく現場目線の例で、試験問題文の読み方に繋げる。大手予備校サイトはコンプライアンス上こうした生々しい具体例を書けないため、ここが個人ブログの差別化ポイントになる
3. **公式の用語への着地**：2つのたとえで温まったところで、実際の試験に出る正式な言葉・仕訳・数式に接続する

例：減価償却＝①「30万円のMacBook Proを5年間使うなら、初年度に30万円まるごと赤字にせず毎年6万円ずつ使ったことにする」→②「バイト先のカフェが100万円のエスプレッソマシンを導入したときも同じ考え方」→③「これを簿記のルールで仕訳にすると…」

### たとえ話以外の独自性の出し方

一次情報（独自性）は例え話だけではない。以下も強力な武器になる：

- **iPadの勉強ノートの実物スクショ**：手書きのまとめ・図解・書き込みをそのまま載せる（個人情報が写っていればモザイク処理）。恥ずかしい場合は、テキスト入力で綺麗にまとめた部分だけ、または矢印・メモだけ手書きにした画面でもよい。ノート全体ではなく該当箇所だけ切り抜けば見せやすい
- **失敗談・つまずきの記録**：模試の点数、間違えた問題、不合格の経験など、綺麗事ではない泥臭い記録。感情の起伏は再現できないため最強の独自性になる
- **リアルなタイムスケジュール**：「1日3時間勉強しましょう」ではなく、大学・アルバイト・開発と両立させている実際の1日の使い方を分単位で
- **勉強を支える環境・道具のレビュー**：ペーパーライクフィルム、Apple Pencilの替え芯、集中用のBGM・おやつなど、実際に使っているものだけを書く

**「体験の証拠」としての画像は必ず実物のスクリーンショットを使う。AI生成画像（Midjourney・DALL-E等）は使わない。** Googleの画像認識は「AI生成っぽい質感」と「本物の使用感（ペンのブレ、フォルダの雑多さ）」を区別しており、AI生成画像はむしろ「AI量産サイト」の疑いを強める逆効果になる。見た目の綺麗さより本物であることを優先する。

一方、**概念を説明するための図解・インフォグラフィック（比較表・フロー図など）は別扱い**。これは「体験の証拠」ではなく説明の補助であり、AI生成でも問題ない。ただし画像生成AIは図中の日本語テキスト（用語・数値）を正確に描画するのが苦手なため、既存の図解と同様にSVGで直接作成する方が用語・数値の正確性を担保できる。

## フェーズ4：編集者パス（客観チェック）

書き上げた後、以下の観点で自分自身にレビューさせる：

1. ネットのまとめサイトと同じに見える退屈なパートはないか
2. 具体的なエピソード（数字・固有名詞・失敗談）をもっと掘り下げられる余地はないか
3. AdSenseの「有用性の低いコンテンツ」に引っかかりそうな薄いセクションはないか
4. 他の記事（特に同じテンプレートを使うシリーズ記事）と構成・言い回しが似すぎていないか
5. **`company/reports/20260714-AdSenseポリシー参考URL一覧.md`の「コンテンツの最小要件」「質の低いコンテンツ」を実際に開いて読み返し、書いた記事がその基準を満たしているか照らし合わせる**（2026-07-15追加。ポリシーを覚えているつもりで参照を省略しない）

## 文体ルール

- 漢字とひらがなのバランス：「〜の時」→「〜のとき」、「〜と言う」→「〜という」など開き気味に
- スマホ表示を意識し、2〜3文ごとに改行を入れる
- [[feedback_article_image_completeness]]：OGP画像・図解をテキストと同時に用意する。後回しにしない

## 前提：一次情報はAIが作れない

「0から1（体験談）を生み出す」ことはできない。取締役の実体験・メモ・写真がないまま執筆フェーズに進まない。一次情報が無い記事テーマは、無理に書かず後回しにする判断もあり。

## テンプレート量産への注意

同一テンプレート（例：「独学合格ガイド」シリーズ）を多数の記事に使い回す場合、構成の型は同じでも、フェーズ1〜2の一次情報が記事ごとに本当に異なる内容になっているか確認する。使い回しの構成に使い回しの言い回しが乗ると、AdSenseに「複製されたコンテンツ」と判定されるリスクがある（[[project_adsense_thin_content_flag]]、2026-07-14に実例あり）。
