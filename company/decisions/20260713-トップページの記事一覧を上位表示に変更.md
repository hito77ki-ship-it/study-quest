# トップページ（index.html）の記事一覧セクションを上位表示に変更

**日付：** 2026年7月13日
**提案者：** 取締役
**決定者：** CEO ジョブズ / 取締役
**ステータス：** 承認・実行済み

## 背景

取締役から「このサイト、アプリがもうメインじゃないから、記事の一覧を上に持っていった方がよくない？」と提案があった。index.htmlの実際のセクション構成を調査したところ、以下の順序になっていた。

nav → hero（アプリ訴求）→ stats → features → jobs一覧 → showcase → benefit → voices（アプリの声）→ founder → story → roadmap → faq → updates → cta → **articles（資格別学習ガイド、id="articles"）**

記事一覧はページの最下部近く、13個のアプリ訴求セクションを経由しないとたどり着けない位置にある。すでにこの問題を暗に認識していたと見える痕跡として、「📚 攻略記事」という浮動ボタン（`#artFloat`）が実装されており、クリックで`#articles`まで強制スクロールする導線が別途用意されていた（＝正規の導線だけでは記事にたどり着きにくいことへの応急処置と読める）。

## リサーチ・検証内容

- **ホームページSEOのベストプラクティス（2026年）：** ファーストビューは「ページの主目的に合った明確な見出し・実際の訪問者が期待する結果」を示すべきとされる。モバイルファーストインデックスの観点では、モバイルのファーストビューが実質的な主要SEO評価対象になる。現状のファーストビューはアプリ訴求のみで、記事に触れていない。
- **AdSense審査観点：** 「トップページを空にしない」「E-E-A-Tを満たす実質的なコンテンツ」が求められる。トップページ自体が空白ではないため直接的な違反ではないが、13セクション連続のアプリ販促コピーは、実質的に「コンテンツサイト」ではなく「SaaSランディングページ」に近い印象をクローラー・審査双方に与えかねない。
- **実データ（[[project_app_vs_content_priority]]、2026-07-08時点）：** アプリ登録23人中、現在活動中のユーザーは0人。記事コンテンツが実質的に唯一の稼働資産になっている。
- **JS依存関係の確認：** `IntersectionObserver`はセレクタベースで動作し、`scrollY`のnav切り替えも閾値ベースのため、セクションの並び順を変えても既存スクリプトへの影響はないことをgrepで確認済み。

## ジョブズの所見

「それはユーザーの人生を本当に変えるか？」という基準で見ると、今のトップページは0人しか使っていないアプリのために13セクション分のスクロールを全訪問者に強いている。これは実データを無視した構成であり、今の実態（記事が主力資産）と完全にズレている。シンプルさの原則にも反する——今のサイトの「本質」は記事なのに、それを一番後回しにしている。

SEO・AdSenseという最優先目標への寄与も明確。ファーストビュー〜スクロール直後の位置に記事一覧を置けば、検索意図とページ構成の一致度が上がり、直帰率の改善（＝エンゲージメント信号の改善）が期待できる。

ただし全面刷新はリスクとコストに見合わない。hero（ブランドの第一印象・H1）と`btn-hero`のCTA自体はそのまま残し、**記事一覧セクションをheroの直後（stats以降の13セクションより前）に移動する**、という最小限かつ効果最大の変更が妥当と判断する。hero見出しの文言変更（「三日坊主になる前に」→記事訴求文言など）は今回のスコープ外とし、必要なら別途提案する。

辛口に言うと、この提案自体は取締役から出てきたが、正直もっと早く気づくべきだった。[[project_focus_shift_articles]]で方針転換したのが7/6頃、実データでアプリの定着率がほぼゼロと判明したのが7/8。それから5日間、トップページの構成だけが古い優先順位のまま放置されていた。

## 実施方針（承認された場合）

1. index.html内の`<section id="articles" ...>`ブロック（1478行目〜2192行目付近）を、`<section class="hero">`の閉じタグ直後・`<section class="stats">`の直前に移動
2. `nav`のリンクに「学習ガイド」等、`#articles`への直接アンカーを追加するかは別途検討（今回は移動のみ実施し、様子を見る）
3. HTMLバランスhookで移動後の構文確認
4. スクリーンショットでレイアウト崩れがないか目視確認（hero直下に記事一覧が正しく表示されるか）
5. コミット・push

## 影響

- 対象：index.htmlのセクション順序のみ（本文・記事データ・JS・CSSは無変更）
- リスク：低（DOM順序変更のみ、JSはセレクタベースで順序非依存と確認済み）
